## Patterns

### E2E Tests

No E2E tests exist. The Cypress setup is boilerplate generated by `cypress open` or `npx cypress install`. The `commands.ts` file contains only commented-out examples from the Cypress documentation with no implementation.

### Component Tests

The `component.ts` support file registers `cy.mount`:

```ts
import { mount } from 'cypress/react'
Cypress.Commands.add('mount', mount)
```

This enables Cypress component-level testing where individual React components can be mounted and asserted against without a running server. However, no component test specs have been written.

## Recommendations

To establish a baseline testing capability with minimal friction:

1. **Unit tests with Vitest:** Add `vitest` + `@testing-library/react` + `@testing-library/user-event`. Start with pure utility functions (`buildPath`, `camelToHumanText`, `formatDDMMYY`, `validateEmail`) and Redux slice reducers (notification, tenant state transitions). Zero mocking required for these.

2. **Hook tests:** `use-suspense-error` and `use-search-bar-params` can be tested with `renderHook` and a mock Redux store without network calls.

3. **Permission logic test:** The `hasEnoughPermissions` logic in `use-profile.tsx` is complex and critical — it is the primary access control gate. A focused unit test covering role/permission combinations would catch regressions.

4. **E2E with Cypress:** Auth0 integration requires either the `cypress-auth0-spa-js-implicit` pattern (programmatic login via Auth0 Management API) or a mock auth provider for test environments. This is non-trivial and should be tackled after unit tests provide a baseline.

5. **Add a `test` script** to `package.json` so `npm test` works as expected by CI/CD pipelines.
